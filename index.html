Projeto GIT

O GIT é um sistema que registras as mudanças feitas durante o processo de ciclo de vida em um arquivo ou conjunto de arquivos.

O projeto teve vida pelas mãos da equipe do Kernel do Linux, que na epóca utilizava o BitKeeper(Controlador de versão da Apache) e por haver necessidade de mais funcionalidades que o mesmo não atendia, foi dada vida há esta excelente ferramenta de controle de versão.

Fundamentos

De inicio o mais importante é entender como o GIT gerência e armazena todas as informações que são incluidas ou alteradas pelo usuario. Fazendo uma comparação simples com outros controladores de versão (Subversion, Perforce, Bazaar, e assim por diante) que tratam as informações como um conjunto de arquivos e as mudanças feitas a cada arquivo ao longo do tempo.

Já o GIT trata seus dados mais como um conjunto de imagens de um sistema de arquivos em miniatura. Toda vez que você fizer um commit, ou salvar o estado de seu projeto no GIT, ele basicamente tira uma foto de todos os seus arquivos e armazena uma referência para esse conjunto de arquivos.

Além de trabalhar com três estados diferentes, sendo a stage que se refere ao pacote atual de desenvolvimento, commit conjunto de mudanças que será entregue e por ultimo push onde o pacote é enviado.

Configuração

git config --global user.name "[usuario]"

Define um nome identificável para associar ao revisar o histórico de versões.

git config --global user.email "[email]"

Define um email que será associado a cada marcador do histórico.

git config --global color.ui auto

Define a coloração automática da linha de comando para o Git para facilitar a revisão.

Inicio de Projeto

git init

Inicializar um diretório existente como um repositório Git.

git clone [url]

recuperar um repositório inteiro de um local hospedado via URL.

Stage & Snapshot

git status

Apresenta os arquivos modificados no diretório ativo, preparado para sua próxima confirmação.

git add [arquivo/diretório]

Adiciona um arquivo como parece agora ao seu próximo commit (Stage).

git reset [arquivo/diretório]

Desvincula um arquivo enquanto mantém as alterações no diretório de trabalho.

git diff

Difere o que foi alterado.

Stage & Snapshot

git diff --staged

Difere o que foi alterado, mas não está comitado.

git commit -m “[mensagem]”

Confirma seu conteúdo na Stage como um novo commit snapshot

Branch & Merge

git branch

Lista todas as branchs e marca(*) a que está atualmente ativa.

git branch [nome]

Cria uma nova branch, a partir do commit atual.

git checkout [branch]

Muda para outra branch.

git merge [branch]

Mescla o histórico da branch especificado no atual.

Inspecione & Compare

git log

Apresenta todos os commits no histórico da branch atual.

git log --follow [arquivo/diretório]

Apresenta os commits que alteraram o arquivo, mesmo entre os renomeados.

git show [SHA]

Apresenta um objeto no Git em formato legível.

Compartilhe & Atualize

git remote add [alias] [url]

Adicione um URL git como um alias.

git fetch [alias]

Busca todas as branchs refentes a informada.

git push [alias] [branch]

Envia a branch local para um repostório remoto.

git pull

Busca e mescla quaisquer commit da branch remota de rastreamento.

Reescrever

git rebase [branch]

Aplica quaisquer commits da branch atual antes da especificada.

git reset --hard [commit]

Limpa toda a stage.

Commits temporários

git stash

Salva as alterações realizada.

git stash list

Lista as alterações realizadas.

git stash drop

Descarta todas as alterações feitas.

WorkFlow

Centralizado

Como isso funciona?

Os desenvolvedores começam clonando o repositório central. Em suas próprias cópias locais do projeto, eles editam arquivos e comitam alterações como fariam com o SVN; no entanto, essas novas confirmações são armazenadas localmente - elas são completamente isoladas do repositório central. Isso permite que os desenvolvedores adiem a sincronização até que estejam em um ponto de interrupção conveniente.

A seguir, é apresentado um exemplo do tipo de cenário no qual Centralized Workflow é usado. O cenário é o de um integrante da equipe fazendo uma publicação das alterações locais para a branch master.

Feature Branch

Como isso funciona?

O Feature Branch Workflow assume um repositório central em que a master representa o histórico oficial do projeto. Em vez de confirmar diretamente em sua branch principal local, os desenvolvedores criam uma nova branch sempre que começam a trabalhar em um novo recurso. As branchs dos recursos devem ter nomes descritivos, como itens de menu animados ou problema. A idéia é dar um objetivo claro e altamente focado a cada filial. O Git não faz distinção técnica entre a branch principal e as branch de recursos, para que os desenvolvedores possam editar, preparar e confirmar alterações em uma branch de recursos.

A seguir, é apresentado um exemplo do tipo de cenário no qual Feature Branch Workflow é usado. O cenário é o de uma equipe fazendo uma revisão de código em uma nova solicitação de recebimento de recurso.

GitFlow

O conjunto de ferramentas git-flow é uma ferramenta de linha de comando real que possui um processo de instalação.

O Git-flow é um invólucro do Git. O comando git flow init é uma extensão do comando git init padrão e não altera nada no seu repositório além de criar ramificações para você.

Como isso funciona?

Develop & Master Branches

Em vez de uma única branch master, esse fluxo de trabalho usa duas branchs para registrar o histórico do projeto. A branch principal armazena o histórico oficial de lançamento e a branch de desenvolvimento serve como branch de integração de novos recursos. Também é conveniente marcar todas os commits na branch master com um número de versão.

GitFlow - Feature Branches

Cada novo recurso deve residir em sua própria branch, que pode ser enviada ao repositório central para backup / colaboração. Mas, em vez de se ramificar da master, as feature branches usam a develop como seu branch pai. Quando um recurso é concluído, ele é mesclado novamente ao desenvolvimento. Os recursos nunca devem interagir diretamente com a master.

GitFlow - Release Branches

Depois que a branch de develop adquiriu recursos suficientes para um release, você divide uma branch do release. A criação dessa branch inicia o próximo ciclo de liberação, portanto, nenhum novo recurso pode ser adicionado após esse ponto - apenas correções de bugs, geração de documentação e outras tarefas orientadas a liberação devem ser realizadas nesta branch. Quando estiver pronto para o envio, a branch de lançamento será mesclada na master e marcada com um número de versão. Além disso, ele deve ser mesclado novamente na develop, o que pode ter progredido desde o início do lançamento.

GitFlow - Hotfix Branches

As branchs de manutenção ou “hotfix” são usadas para corrigir rapidamente as liberações de produção. As branchs de hotfix são muito parecidas com as branchs de release e as de feature, exceto que são baseadas na master, e não no desenvolvimento. Esse é a única branch que deve se separar diretamente da master. Assim que a correção estiver concluída, ela deverá ser mesclada na master e na develop e a master deverá ser marcada com um número de versão atualizado.

Fluxo final de trabalho

1° - Uma branch de desenvolvimento é criada a partir da branch master.

2° - Uma branch de release é criada a partir da branch develop.

3° - As feature branchs são criadas a partir da branch develop.

4° - Quando um recurso é concluído, ele é mesclado na branch develop.

5° - Quando a release branch é concluída, ela é mesclada na develop e na master.

6° - Se um problema na master for detectado, uma branch de hotfix será criada a partir da master.

7° - Depois que o hotfix é concluído, ele é mesclado para develop e master.
